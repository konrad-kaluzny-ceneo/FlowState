# F-14: Product voice contract

- **Outcome:** (foundation) bilingual EN/PL product language foundation split into two clearly separated phases: first a technical two-language system, then the target voice/copy contract. The slice should make the FlowState promise legible in both languages without mixing localization infrastructure work with product-copy decisions.
- **Change ID:** product-voice-contract
- **Linear:** [FLO-93](https://linear.app/flowstate-10xdev/issue/FLO-93/flowstate-product-voice-contract-f-14)
- **GitHub:** [#171](https://github.com/konrad-kaluzny-ceneo/FlowState/issues/171)
- **PRD refs:** Secondary success craft; US-03 narrative tone
- **Unlocks:** Language-selection foundation for all future UI work; authoritative copy for S-40 home IA, S-42 day memory, S-41 rail copy
- **Prerequisites:** S-21, S-30, F-04, F-06
- **Parallel with:** —
- **Blockers:** —
- **Acceptance:**
  - **Phase 1 — bilingual system (technical/i18n specialization):**
    - English and Polish are first-class supported languages, not ad-hoc string variants.
    - A clear translation resource structure exists for current and future UI copy.
    - Language selection/defaulting is explicit enough for implementation planning and test planning.
    - Existing user-facing copy migration is scoped as infrastructure work only; no target wording decisions are smuggled into this phase.
  - **Phase 2 — target copy and voice contract (product/copy specialization):**
    - `context/foundation/product-voice.md` — promise, tone, preferred vocabulary, copy zones, examples/non-examples for both EN and PL.
    - Maintainer 5-second purpose test: user answers "what does this app help me decide?" with "what to do next / co teraz".
    - First suggestion includes calm rationale aligned with the voice contract in both languages.
    - Recap closure language covers done / remains / return-to in both languages.
    - Acceptance checklist future slices must cite before changing user-facing copy.
  - The implementation plan must assign Phase 1 and Phase 2 to separate sub-agent tracks or clearly distinct workstreams because the skills differ: localization architecture vs product copy design.
- **Unknowns:**
  - Exact language defaulting policy and user-facing language switch placement — owner: implementer/user. Block: no.
  - Whether Phase 1 should use a lightweight local dictionary or introduce a formal i18n library — owner: implementer during research. Block: no.
  - Migration order for existing English labels — owner: implementer in Phase 1. Block: no.
- **Risk:** Blending localization infrastructure with copywriting would create noisy scope, weak ownership, and inconsistent results. Partial copy migration before contract acceptance can mix Polish and English labels, while generic wellness language weakens product trust. Expand score 9.6.
- **Status:** done
